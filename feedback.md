## 🟡 종합 평가: 범위 조정 후 최소 작동 MVP는 현실적으로 완주 가능

**핵심**: 하루 1시간 × 3주(21시간)로 **팀원 → 팀장 단계의 기본 E2E 플로우는 충분히 가능**.
다만, 예외처리·테스트·부장~임원 검증은 제한적일 것. 초기 설계를 generic으로 하면 추후 계층 확장이 간단함.

## ✅ 잘한 점

### 1. **복잡한 시스템 경험 있음** (증권수탁 차세대 시스템)
   - Java/JavaScript/Oracle 기반의 복잡한 엔터프라이즈 프로젝트 경험
   - 기술 습득력과 문제 해결 능력이 있을 것으로 예상

### 2. **모든 리소스를 활용한 자력 학습 가능**
   - 공식 문서, 검색, AI 등 모든 방법을 활용 가능
   - 막힌 부분을 빠르게 해결할 수 있는 태도

### 3. **개발 환경 제약 없음**
   - 외부 API 호출, npm 패키지 설치 가능
   - 기술적 자유도가 높음

## ⚠️ 부족한 점 (완주 위협 순서)

### 1. **시간 여전히 부족** (21시간은 최소선)
   - **필요시간**: generic 설계 + 팀원→팀장 E2E + 기본 테스트 = 약 40~50시간
   - **가능시간**: 21시간 (약 42~52% 수준)
   - **대응**: 매주 계획을 철저히 하고, 예외처리·고급 기능은 과감히 빼기

### 2. **모든 기술이 처음** (Next.js, TypeScript, Supabase)
   - git/PR/Claude Code 경험도 없음 → 개발 워크플로우 학습 필요
   - 첫 1주는 기술 학습으로 시간이 많이 걸릴 가능성
   - **대응**: 튜토리얼/예제로 빠르게 기초 잡기, 초기 구조 설계에 집중

### 3. **HWP 자동 생성 + 오탈자 검출은 추가 기술부채**
   - LibreOffice CLI (2~3시간), hunspell (1~2시간) 통합 필요
   - 예상치 못한 버그 발생 시 시간 소모 가능
   - **대응**: 라이브러리는 안정적인 버전 선택, 초기에 검증해두기

## 🔄 범위 조정 내용 (최종 확정)

### ✅ 추가된 기능
- **오탈자 검출** (hunspell): 팀원 입력폼 + 팀장 에디터 모드에서 실시간 검출
- **HWP 실시간 미리보기**: 입력/수정 중 오른쪽에 최종 보고서 형태 표시
- **HWP 자동 생성**: docx.js + LibreOffice CLI로 DOCX → HWP 변환

### 🔧 단순화된 기능
- **문구 수정**: 각 계층에서 수정한 버전을 상위로 전달 (원문 비교 X, 이력 추적 X)
- **기술스택**: 라이브러리는 나중에 교체 가능하므로 hunspell로 가볍게 시작

### 📌 아키텍처 전략
- **초기 설계를 generic하게**: 팀원→팀장 로직을 범용으로 만들면, 부장~임원도 자동으로 확장 가능
- **테스트 우선순위**: 팀원→팀장 완벽하게, 부장~임원은 코드만 구현 후 추후 검증
